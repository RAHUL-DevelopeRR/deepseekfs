# DeepSeekFS Architecture Report

## 1. High-Level Overview
DeepSeekFS is a monolithic, local-first semantic file search engine. It operates completely offline, indexing actual file contents rather than relying solely on filenames or metadata tags. The application uses HuggingFace's `SentenceTransformer` (`all-MiniLM-L6-v2`) running on the CPU to create dense vector representations of documents, which are stored and queried using FAISS.

## 2. Dual-Interface Architecture
The system employs a dual-entry point architecture, allowing the application to run either as a web server or as a native desktop application. 

### A. Web API & Lightweight UI (`run.py` & `app/`)
- **API Server:** Built with FastAPI (`app.main.py`). Starts a Uvicorn server exposing endpoints for searching, indexing, and health checks on port 8000.
- **Routes:** The `api/` directory contains `routes` and `schemas` for the FastAPI backend.
- **Web UI:** `run.py` uses `pywebview` to spawn a lightweight window pointing to an internally served HTML page, communicating with the FastAPI backend over HTTP. It can gracefully degrade to a headless API server if the GUI cannot start.

### B. Native Desktop Application (`run_desktop.py`)
- **PyQt6 Monolith:** A single executable native application that bypasses the HTTP API and network sockets entirely.
- **Service Layer Facade:** It communicates directly with the business logic via `services/desktop_service.py` (`DesktopService`).
- **Threading:** Uses PyQt background threads (`IndexThread`, `SearchThread`) to ensure the UI remains responsive during heavy local parsing and FAISS vector calculations.

## 3. Core Domain Modules (`core/`)
The `core/` folder encapsulates all domain logic and system operations:
- \`ingestion/\`: Specifically the \`FileParser\`, utilizes libraries like \`PyMuPDF\`, \`python-docx\`, etc. to extract textual content from user documents (up to 5000 chars), handling formats like PDFs, Docs, TXT, CSV, and code files.
- \`embeddings/\`: Wraps the Hugging Face `SentenceTransformer` model to convert the extracted text into 384-dimensional dense vectors.
- \`indexing/\`: Manages connection and transactions to a local **FAISS** index, persisting both document vectors and metadata (`metadata.pkl`).
- \`search/\`: Orchestrates retrieval by generating vectors for user queries and performing K-Nearest-Neighbor Euclidean distance calculations on the FAISS index. It includes a custom ranking formula balancing semantic similarity, time decay, and frequency scores.
- \`watcher/\` & \`time/\`: Manages directory monitoring using background watchdog threads to invoke incremental scans when new files appear.

## 4. System Workflows

### Initial Execution & Ingestion
1. The user launches `run.py` or `run_desktop.py`.
2. The application detects default target folders (Documents, Downloads, Desktop) or parses `config.py`.
3. An initial full scan loops through files, the \`FileParser\` extracts text, the \`Embedder\` transforms it to vectors, and the documents are saved to the FAISS index.
4. Watchdog observers are attached to directories for real-time changes.

### Query Execution
1. The user inputs a query via the PyQt6 UI or the Web Interface.
2. The query is passed to the search service (HTTP API or DesktopService).
3. The embedding model generates a 384D query vector.
4. FAISS retrieves the most mathematically similar document vectors.
5. The ranker adjusts scores computationally (e.g., adding bonus points for recently modified files).
6. The final mapped results form the output display.

## 5. Summary
DeepSeekFS implements a classical monolith cleanly separating the **Application/Transport Layer** (`app/`, `api/`, `run.py`, `run_desktop.py`), **Service Layer** (`services/`), and **Domain/Infrastructure Layer** (`core/`). This decoupling makes it resilient and easy to package while allowing it to support multiple execution contexts (Web vs. Native Desktop) using the same computational backbone.
